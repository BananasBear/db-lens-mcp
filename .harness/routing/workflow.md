# Workflow

## 需求分析

1. 识别任务类型和关键词。
2. 读取 `context-map.md` 中匹配的文档。
3. 列出已确认事实、未知项和建议假设。
4. 对影响安全、部署或架构的事项，先向用户确认。

## 方案设计

1. 给出 2 到 3 个可选方案。
2. 说明每个方案的复杂度、风险和适用场景。
3. 明确推荐方案。
4. 用户确认后再写入 `.harness/docs/` 中对应文档。

## 实现

1. 先读取相关整体设计、模块设计和规范。
2. 保持改动聚焦，不做无关重构。
3. 新增高风险能力时先补安全规则。
4. 优先补最小相关测试。

## 收尾

1. 说明改动文件。
2. 说明验证命令和结果。
3. 说明是否改动业务代码、CI、API、数据库或 Git。
4. 判断是否需要写入 `.harness/changes/`。
