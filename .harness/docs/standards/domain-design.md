# Domain Design Standards

## 核心原则

- 遵循领域模型设计思想，围绕真实业务场景和业务语言组织代码。
- 不得为了技术形式强行抽象领域对象；领域模型必须服务于当前业务问题。
- 不确定的业务规则、数据含义和边界条件必须显式标记为“待确认”或“信息不足”。

## 领域边界

- 不同领域之间保持清晰边界，通过明确接口交互。
- 跨领域调用应依赖稳定的应用服务或领域接口，不直接访问对方内部实现。
- 数据库访问、配置加密、MCP 协议、安全校验、风险分析应保持职责独立。
- 模块内部可以封装实现细节，但不要隐藏关键业务规则。

## 模块封装

- 模块接口应小而明确，参数和返回值表达业务含义。
- 模块应便于替换和扩展，例如后续增加 PostgreSQL 时，不应改动 MCP 工具层的大量逻辑。
- 共享能力要在出现真实重复和清晰边界后再抽象。
- 避免为了“未来可能需要”提前引入复杂基类、插件系统或深层继承。

## 领域对象

- 领域对象应表达项目核心概念，例如连接配置、表结构、索引、执行计划、风险提示。
- 领域对象不应直接承担外部协议适配、数据库驱动调用或命令行解析职责。
- 领域对象字段必须来自真实业务需要或已确认的数据库元数据，不得凭空添加。

## 交互规范

- MCP 工具层负责协议输入输出，不直接拼接数据库查询细节。
- 应用服务层负责组织业务流程，例如“查询相关表结构 + 索引 + EXPLAIN + 风险提示”。
- 领域层负责表达概念和规则，例如 SQL 安全判断、风险提示规则。
- 基础设施层负责数据库驱动、配置加载、密钥读取和日志输出。

## 代码可维护性

- 代码结构应让开发人员能快速找到领域入口、模块边界和安全拦截点。
- 新增模块时，应同步更新 `.harness/routing/context-map.md` 和对应模块设计文档。
- 当业务规则变化时，优先更新领域模型和模块文档，再调整实现。
