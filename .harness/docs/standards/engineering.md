# Engineering Standards

## 技术栈

- 第一版使用 Python。
- 包管理和本地运行优先考虑 uv/uvx。
- 服务器部署优先使用 Docker。

## 工程原则

- 不得编造业务逻辑、业务字段、状态含义、外部接口或系统能力。
- 需求、代码或文档无法确认的内容，必须明确标记为“待确认”或“信息不足”，并说明建议假设。
- 代码应简洁清晰，避免过度设计、过度封装和提前抽象。
- 协议入口、应用服务、领域安全规则、基础设施分层。
- 数据库访问能力不要直接散落在 MCP 工具处理函数中。
- 所有外部输入必须显式校验。
- 错误消息应说明原因和建议处理方式，避免泄露敏感信息。
- 模块封装应高效简洁，优先暴露稳定、必要、可测试的接口。
- 代码风格以开发人员阅读和排查为优先，命名应表达业务含义。

## 代码可读性

- 函数和类应保持单一职责，避免文件和函数过长。
- 优先使用直接、可读的控制流；复杂技巧必须有明确收益。
- 命名使用领域语言，避免只有实现细节的缩写。
- 错误处理路径要清晰，不吞异常，不返回含糊结果。
- 配置、安全、数据库访问等边界代码要容易定位和审查。

## 注释

- 业务规则、领域约束、安全边界、非显然算法和重要取舍必须写注释。
- 注释应解释“为什么这样做”和“边界是什么”，不要机械重复代码正在做什么。
- 临时假设必须在注释或文档中标记“待确认”，并说明影响范围。
- 涉及风险拦截、SQL 校验、密钥处理、脱敏输出的代码，应有清晰注释说明安全意图。

## 日志

- 默认不记录完整数据库密码、密钥、连接串。
- SQL 日志需要考虑脱敏和可配置。
- 服务器模式应至少记录请求时间、工具名、连接配置别名和结果状态。

## 待确认

- Python 版本下限。
- 代码格式化和 lint 工具。
- 日志库和结构化日志格式。
