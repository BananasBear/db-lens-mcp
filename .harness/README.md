# Project Harness

`.harness/` 是本项目 AI 开发的共享事实源，用于沉淀上下文路由、整体设计、模块设计、项目规划、开发规范和变更记录。

## 目标

- 让 AI 在开发前知道应该读取哪些上下文。
- 将项目整体设计、模块设计、安全边界和开发规范放在稳定位置。
- 避免把长期规则散落在对话、临时计划或入口文件中。
- 支持从空项目逐步演进为可维护的 MCP 工程。

## 规则优先级

1. 用户在当前对话中的明确要求。
2. `AGENTS.md` 中的项目级协作规则。
3. `.harness/` 中的长期规则和上下文。
4. 代码、测试和配置中的现有事实。

如果文档与代码冲突，应优先核验代码，并在回复中说明冲突。

## 读取顺序

1. `AGENTS.md`
2. `.harness/README.md`
3. `.harness/routing/context-map.md`
4. `.harness/routing/workflow.md`
5. `.harness/docs/` 下与任务相关的文档
6. `.harness/changes/README.md`

## 目录说明

```text
.harness/
  routing/                  # 路由层：任务分类、上下文索引、工作流
  docs/                     # 项目文档：整体设计、规划、模块设计、规范
    architecture.md         # 项目整体设计
    roadmap.md              # 项目规划
    modules/                # 不同功能模块设计
    standards/              # 开发规范和规则约束
  changes/                  # 重要变更记录
  templates/                # 可复用模板
```

## 标准工作流

1. 先匹配任务类型和关键词。
2. 读取相关上下文和已确认设计。
3. 明确事实、未知项和假设。
4. 先给方案或计划，再实现。
5. 实现时保持最小改动。
6. 收尾时说明改动、验证、风险和文档影响。

## 当前阶段

项目处于早期设计阶段。当前文档只记录已确认的产品方向和协作规则；具体代码结构、依赖选择、部署脚本、MCP 工具接口仍待进一步设计。
